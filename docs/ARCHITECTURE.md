# Star Frame Architecture

How Star Frame works internally — from entrypoint to instruction execution.

## High-Level Overview

Star Frame is a **trait-based** framework. Every component — programs, instructions, accounts, validation — is defined by traits. The framework uses Rust's type system and proc macros to generate the glue code, giving you compile-time safety with minimal runtime overhead.

```
Transaction arrives
    │
    ▼
┌──────────────────────────────┐
│  StarFrameProgram::entrypoint│  ← Generated by #[derive(StarFrameProgram)]
│  (program_id, accounts, data)│
└──────────┬───────────────────┘
           │
           ▼
┌──────────────────────────────┐
│  InstructionSet::dispatch    │  ← Generated by #[derive(InstructionSet)]
│  (match discriminant → Ix)   │
└──────────┬───────────────────┘
           │
           ▼
┌──────────────────────────────┐
│  Instruction::process_from_raw│  ← Generic impl for StarFrameInstruction
│  1. BorshDeserialize data    │
│  2. InstructionArgs::split   │
│  3. AccountSet::decode       │
│  4. AccountSet::validate     │
│  5. StarFrameInstruction::run│
│  6. AccountSet::cleanup      │
│  7. Set return data          │
└──────────────────────────────┘
```

## The Instruction Lifecycle

### 1. Program Entrypoint

```rust
#[derive(StarFrameProgram)]
#[program(
    instruction_set = MyInstructionSet,
    id = "MyProgramId..."
)]
pub struct MyProgram;
```

The `StarFrameProgram` derive macro:
- Implements the `StarFrameProgram` trait (with `ID`, `InstructionSet`, `entrypoint`)
- Generates the Solana `entrypoint!` macro call
- Creates `StarFrameDeclaredProgram` type alias, `ID` const, `check_id()`, `id()` helpers
- Sets up `program_setup!` for top-level program utilities

The entrypoint simply delegates to `InstructionSet::dispatch`.

### 2. Instruction Dispatch

```rust
#[derive(InstructionSet)]
pub enum MyInstructionSet {
    Initialize(Initialize),
    Increment(Increment),
}
```

The `InstructionSet` derive macro:
- Generates **8-byte discriminants** for each variant (Anchor-compatible sighash by default)
- Implements `dispatch()` which reads the first 8 bytes, matches them, and calls the appropriate `Instruction::process_from_raw`

**Discriminant format:** By default, uses `sha256("global:<snake_case_name>")[..8]` — the same as Anchor's sighash. You can override discriminants with `#[discriminant(...)]`.

### 3. Instruction Data Parsing

```rust
#[derive(BorshSerialize, BorshDeserialize, Debug, InstructionArgs)]
pub struct Initialize {
    #[ix_args(run)]         // passed to process()
    pub start_at: u64,
    #[ix_args(&validate)]   // passed to validate_accounts() by reference
    pub some_config: u8,
}
```

`InstructionArgs` splits deserialized data into four lifecycle phases:
- **`decode`** — arguments needed during account decoding
- **`validate`** — arguments needed during account validation
- **`run`** — arguments passed to `StarFrameInstruction::process`
- **`cleanup`** — arguments needed during cleanup

Use `#[ix_args(run)]` to tag fields. By default, fields with no tag are not passed to any phase. You can pass by reference with `#[ix_args(&run)]` or `#[ix_args(&validate)]`.

### 4. Account Set Decode

```rust
#[derive(AccountSet)]
pub struct InitializeAccounts {
    pub authority: Signer<Mut<SystemAccount>>,
    pub counter: Init<Seeded<Account<CounterAccount>>>,
    pub system_program: Program<System>,
}
```

**Decoding** consumes `AccountInfo`s from the remaining accounts slice, one per field (in order). Each account type implements `AccountSetDecode` which pops from the front of the accounts slice.

### 5. Account Set Validation

Validation runs after all accounts are decoded. This is where the real magic happens.

Each account type has layered validation through modifiers:

```
Init<Seeded<Account<CounterAccount>>>
  │     │      │         │
  │     │      │         └─ ProgramAccount: validates owner + discriminant
  │     │      └─ Account: wraps AccountInfo with typed data access
  │     └─ Seeded: validates PDA matches expected seeds
  └─ Init: creates the account (system_create_account + init data)
```

The `#[validate(...)]` attribute on fields controls what arguments are passed to each account's validation:

```rust
#[validate(funder)]                           // Tags this as the rent funder (cached in Context)
#[validate(recipient)]                        // Tags as the rent recipient
#[validate(arg = Create(()))]                 // Passes Create(()) to Init's validation
#[validate(arg = Seeds(MySeeds { ... }))]     // Passes seeds to Seeded's validation
#[validate(address = &some_pubkey)]           // Validates pubkey matches
#[validate(extra_validation = self.check())]  // Custom validation expression
```

**Validation argument resolution:** The `arg = (A, B)` syntax creates a tuple that gets destructured across the modifier stack. `Init` takes the first element, `Seeded` takes the second, etc.

### 6. Instruction Processing

```rust
#[star_frame_instruction]
fn Initialize(accounts: &mut InitializeAccounts, start_at: u64) -> Result<()> {
    // Your logic here
    Ok(())
}
```

The `star_frame_instruction` macro implements `StarFrameInstruction` for you. The function name must match the instruction struct name. Parameters after `accounts` come from `InstructionArgs::split_to_args`.

### 7. Account Set Cleanup

After instruction execution, cleanup runs in reverse. This handles:
- **Rent normalization** — adjusting account lamports for size changes
- **Account closing** — zeroing data and transferring lamports
- **Custom cleanup** via `#[cleanup(arg = ...)]`

```rust
#[derive(AccountSet)]
pub struct CloseAccounts {
    #[validate(recipient)]
    pub funds_to: Mut<SystemAccount>,
    #[cleanup(arg = CloseAccount(()))]    // Close and send lamports to recipient
    pub account: Mut<Account<MyAccount>>,
}
```

## Account Type System

Star Frame's account types compose through wrapping:

### Base Types

| Type | Description |
|------|-------------|
| `AccountInfo` | Raw unchecked account |
| `SystemAccount` | Validates owner is System Program |
| `Account<T>` | Validates owner + discriminant, provides typed `data()`/`data_mut()` |
| `BorshAccount<T>` | Like Account but uses Borsh instead of zero-copy |
| `ValidatedAccount<T>` | Account with custom `AccountValidate` logic |
| `Program<T>` | Validates pubkey matches `T::ID` |

### Modifiers (wrappers)

| Modifier | Effect |
|----------|--------|
| `Signer<T>` | Validates account is a signer |
| `Mut<T>` | Validates account is writable |
| `Init<T>` | Creates the account during validation |
| `Seeded<T>` | Validates PDA address matches seeds |

### Composition Examples

```rust
Signer                              // Just a signer (AccountInfo underneath)
Signer<SystemAccount>               // Signed system account
Signer<Mut<SystemAccount>>          // Signed, writable system account
Mut<Account<MyData>>                // Writable typed account
Init<Seeded<Account<MyData>>>       // Init a PDA-derived typed account
Init<Signer<Account<MyData>>>       // Init a keypair-signed account
```

**Order matters:** `Signer<Mut<T>>` and `Mut<Signer<T>>` both work, but follow convention: `Signer` on the outside for signers, `Mut` wrapping the inner type.

## PDA / Seeds System

PDAs are handled through the `GetSeeds` trait and `Seeded` modifier:

```rust
#[derive(Debug, GetSeeds, Clone)]
#[get_seeds(seed_const = b"COUNTER")]
pub struct CounterSeeds {
    pub authority: Pubkey,
}
```

This generates: `seeds = [b"COUNTER", authority.as_ref(), &[bump]]`

The `Seed` trait is blanket-implemented for any `NoUninit` type via `bytemuck::bytes_of`. The last element of `seeds()` is always `&[]` — a placeholder that gets replaced with the bump byte.

**Linking seeds to accounts:**

```rust
#[derive(ProgramAccount)]
#[program_account(seeds = CounterSeeds)]
pub struct CounterAccount { ... }
```

This implements `HasSeeds` for `Account<CounterAccount>`, enabling `Seeded<Account<CounterAccount>>` to automatically know its seed type.

## Zero-Copy Data Model

Star Frame uses `bytemuck` for zero-copy account data:

```rust
#[zero_copy(pod)]
#[derive(ProgramAccount, Default, Debug, Eq, PartialEq)]
pub struct CounterAccount {
    pub authority: Pubkey,
    pub count: u64,
}
```

`#[zero_copy(pod)]` expands to `#[derive(Align1, Pod, Zeroable, Copy, Clone)]` with `#[repr(C, packed)]`. The `Align1` trait ensures 1-byte alignment (required for Solana accounts which don't guarantee alignment).

### Account Discriminants

Each `ProgramAccount` gets a discriminant — an 8-byte value written at the start of account data. By default, this uses the Anchor-compatible format: `sha256("account:<TypeName>")[..8]`.

The discriminant is validated on every account access and written during `Init`.

## Context

The `Context` struct provides shared state during instruction execution:

```rust
pub struct Context {
    program_id: &'static Pubkey,
    rent_cache: Cell<Option<Rent>>,    // Lazy-loaded, cached
    clock_cache: Cell<Option<Clock>>,  // Lazy-loaded, cached
    recipient: Option<Box<dyn CanAddLamports>>,  // Set by #[validate(recipient)]
    funder: Option<Box<dyn CanFundRent>>,         // Set by #[validate(funder)]
}
```

Key features:
- **Sysvar caching** — `ctx.get_rent()` and `ctx.get_clock()` load once, cached for the instruction
- **Funder/Recipient** — `#[validate(funder)]` and `#[validate(recipient)]` cache accounts for automatic rent management

## Unsized Type System

For variable-length accounts (dynamic arrays, maps), Star Frame provides `unsized_type`:

```rust
#[unsized_type(program_account, seeds = MarketSeeds)]
pub struct Market {
    pub version: u8,
    pub authority: Pubkey,
    #[unsized_start]
    pub bids: OrderBookSide,
    pub asks: OrderBookSide,
}
```

Fields before `#[unsized_start]` are fixed-size (bytemuck). Fields after are dynamically-sized using Star Frame's custom DSL with types like `List<T>`, `Map<K, V>`, `UnsizedMap<K, V>`.

Unsized types support operations like `insert`, `remove`, `binary_search`, and automatically handle account reallocation.

## IDL Generation

Star Frame generates [Codama](https://github.com/codama-idl/codama) IDL files:

```rust
#[cfg(feature = "idl")]
#[test]
fn generate_idl() -> Result<()> {
    let idl = StarFrameDeclaredProgram::program_to_idl()?;
    let codama_idl: ProgramNode = idl.try_into()?;
    let idl_json = codama_idl.to_json()?;
    std::fs::write("idl.json", &idl_json)?;
    Ok(())
}
```

The IDL describes instructions, accounts, types, and seed derivations — enabling automatic client generation via Codama tooling.

## Client / CPI Generation

The `AccountSet` derive macro also generates:
- **`*ClientAccounts`** structs for building instructions off-chain
- **`*CpiAccounts`** structs for cross-program invocations

```rust
// Generated automatically from InitializeAccounts:
let ix = MyProgram::instruction(
    &Initialize { start_at: 0 },
    InitializeClientAccounts {
        authority: authority_pubkey,
        counter: counter_pubkey,
        system_program: None,  // Program<T> fields default to T::ID
    },
)?;
```

For CPI:
```rust
MyProgram::cpi(
    MyInstruction { .. },
    MyInstructionCpiAccounts { source, dest, owner },
    None,  // Optional program ID override
).invoke()?;
```
